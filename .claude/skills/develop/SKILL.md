# reviewit.nvim 開発ワークフロー

このスキルは reviewit.nvim の開発を計画からPR作成まで一貫して行います。
計画フェーズでは ultrathink を用いて深い分析を行い、見落としを最小化します。

## 現在のリポジトリ状態

- ブランチ: !`git branch --show-current`
- Git状態: !`git status --short`
- 直近コミット: !`git log --oneline -5`

## フェーズ間のユーザー確認ルール

- **確認必須**: Phase 1（計画）、Phase 2（ブランチ作成）、Phase 7（レビュー）、Phase 8（コミット・PR）
- **自律実行可**: Phase 3（実装）、Phase 4（テスト）、Phase 5（ドキュメント）、Phase 6（品質チェック）
  - ただし問題が発生した場合や判断に迷う場合はユーザーに確認すること

## ワークフロー

### Phase 1: 要件整理と計画 [確認必須]

1. ユーザーの要望 `$ARGUMENTS` を受け取る
2. CLAUDE.md のアーキテクチャセクションを参照し、関連モジュールを特定する
3. 特定した関連モジュールのソースコードを読み、現在の実装を理解する。具体的には:
   - 変更対象モジュールのソースコード全体
   - 変更対象モジュールが依存しているモジュールの公開インターフェース（関数シグネチャと呼び出し箇所）
   - 変更対象モジュールに対応する既存テストファイル
4. 以下の観点で計画を整理し、ユーザーに提示する:
   - **目的**: 何を実現するか
   - **影響モジュール**: どのファイルを変更するか（CLAUDE.md のモジュール責務と照合）
   - **実装方針**: どのように実装するか（既存パターンに従う）
   - **テスト方針**: どの純粋関数をテストするか
   - **ドキュメント更新**: doc/reviewit.txt / README.md / CLAUDE.md のうちどれを更新するか、更新する内容の概要
   - **懸念事項・リスク**: 既存機能への影響、エッジケース
   - **代替案**: 別のアプローチがあれば提示
5. 変更規模が大きい場合（影響モジュールが4つ以上、または新規モジュール追加を含む場合）は複数PRへの分割を提案する
6. ユーザーとの対話で計画を洗練させる（合意に至るまで繰り返す）

**重要**: 計画段階では絶対にコードを書かないこと。Read/Grep/Glob のみ使用する。

### Phase 2: ブランチ作成 [確認必須]

計画合意後：

1. 作業ツリーの状態を確認する
   - クリーンな場合: そのまま続行
   - 未コミットの変更がある場合: 変更内容をユーザーに報告し、stash するか commit するか指示を仰ぐ。勝手に変更を破棄しないこと
2. main ブランチを最新に更新する: `git fetch origin && git checkout main && git pull origin main`
3. 変更の種類に応じたブランチを作成する:
   - 機能追加: `feat/<簡潔な説明>` (例: `feat/add-review-filter`)
   - バグ修正: `fix/<簡潔な説明>` (例: `fix/comment-navigation-wrap`)
   - リファクタリング: `refactor/<簡潔な説明>`
   - ドキュメント: `docs/<簡潔な説明>`
4. ブランチ名はユーザーに提案し、確認を取ってから作成する

### Phase 3: 実装 [自律実行可]

1. 計画に従いコードを実装する
2. 既存のコーディングパターンに従うこと:
   - 非同期処理: `vim.system()` コールバック + `vim.schedule()`
   - 状態管理: `config.state` に集約
   - 純粋関数: テスト可能なロジックは `build_*`, `find_*`, `parse_*`, `format_*`, `should_*`, `make_*`, `calculate_*` として抽出
   - 名前空間: `"reviewit"` を使用
3. 一連の関連する編集が完了した区切りごとに `stylua lua/ plugin/` でフォーマットする（1ファイルごとではなく、論理的なまとまりごと）
4. 実装完了後、既存テストを実行して既存機能を壊していないか確認する:
   ```
   make test
   ```

### Phase 4: テスト [自律実行可]

1. 変更対象モジュールの既存テストファイル（`tests/reviewit/*_spec.lua`）を読み、テストのスタイル・構造・ヘルパーの使い方を把握してから新規テストを書く
2. 新規・変更した純粋関数に対するテストを `tests/reviewit/` に追加する
3. テストファイルの命名: 対象モジュールに対応する `*_spec.lua`
4. テストパターン: busted の `describe`/`it`/`assert` を使用
5. テスト不要の判断基準: 以下の**全て**を満たす関数はテスト不要
   - 条件分岐やデータ変換を含まない
   - vim API または外部コマンドを1つ呼ぶだけ
   - 戻り値の加工をしない
   - 例: `vim.api.nvim_buf_set_lines()` を呼ぶだけのラッパー
6. テスト追加後 `stylua lua/ plugin/ tests/` でフォーマットし、テストを実行:
   ```
   make test
   ```

### Phase 5: ドキュメント更新 [自律実行可]

以下のドキュメントそれぞれについて更新が必要か判断し、必要なものを更新する。

- **doc/reviewit.txt**: Neovim ヘルプドキュメント。コマンド・設定オプション・キーマップ・関数の追加/変更/削除時は**必ず更新**する。既存エントリのフォーマットに合わせること。更新後は `nvim --headless -c "helptags doc/" -c q` でタグを再生成する。
- **README.md**: ユーザー向けドキュメント。新機能・設定オプション・キーマップの追加時に更新する。
- **CLAUDE.md**: アーキテクチャに影響する変更（新モジュール追加、モジュール責務の変更）時に更新する。

いずれのドキュメントも更新不要と判断した場合はその理由をユーザーに説明すること。

### Phase 6: 品質チェック [自律実行可]

すべてのチェックを実行し、全てパスすることを確認する:

```
make all
```

これは以下を順次実行する:
- `luacheck lua/ plugin/ tests/` (lint)
- `stylua --check lua/ plugin/ tests/` (format check)
- `bash run_tests.sh` (tests)

**全チェックがパスするまで次に進まないこと。**

失敗時の対応:
1. エラー内容を分析し、修正を試みる
2. 修正後 `make all` を再実行する
3. **3回連続で失敗した場合**: エラー内容をユーザーに提示し、対処方針について相談する。自力での修正を続けないこと

### Phase 7: 変更レビュー [確認必須]

1. `/review` スキルを使用して変更内容をレビューする
2. レビュー結果をユーザーに提示する:
   - 変更ファイル一覧と各ファイルの変更行数
   - 各変更の概要
   - 潜在的な問題点（正確性、既存機能との整合性、エッジケース）

### Phase 8: コミットとPR作成 [確認必須]

ユーザーの承認後：

1. **コミット分割計画をユーザーに提示する**。以下のルールで分割する:
   - 1コミット = 1つの論理的変更
   - 分割単位（優先度順）:
     1. コア実装の変更（1モジュール1コミットが目安。密結合した複数モジュールの変更は1コミットにまとめてよい）
     2. テストの追加（対応する実装コミットとは別にする）
     3. ドキュメントの更新（1コミットにまとめる）
   - コミットメッセージ: Conventional Commits 形式 (`feat:`, `fix:`, `refactor:`, `test:`, `docs:`)
2. ユーザーの承認後にコミットを実行する
3. draft PRを作成する。PR本文は [pr-template.md](pr-template.md) に従う
4. PR作成後、URLをユーザーに共有する

## 注意事項

- 破壊的な操作（force push, reset --hard, checkout ., clean -f 等）は絶対に行わないこと
- `config.state` への新しいフィールド追加時は `reset_state()` での初期化も忘れないこと
